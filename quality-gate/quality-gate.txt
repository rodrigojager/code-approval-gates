ALTERAÇÃO PONTUAL DO PLANO — quality-check como alias/wrapper Docker
=====================================================================

Objetivo desta alteração
------------------------

Este arquivo altera apenas a parte do plano anterior relacionada ao comando local `quality-check .`.

A decisão agora é:

  O comando `quality-check .` DEVE executar a análise completa usando o Docker sidecar por baixo.

Ou seja, `quality-check .` não deve ser interpretado como uma versão nativa simplificada.
Ele deve ser um alias/wrapper local para `docker run`, montando a pasta atual dentro do container e executando todas as ferramentas empacotadas na imagem `quality-sidecar`.

Adendo — privacidade opt-in
---------------------------

Além da regra container-first, a solução agora também deve seguir estas decisões:

  - PII e secrets NÃO são checks fixos por padrão.
  - PII deve ser verificado somente quando o usuário passar `--enable-pii`.
  - Secrets devem ser verificados somente quando o usuário passar `--enable-secrets`.
  - `--allow-pii` e `--allow-secrets` apenas liberam findings depois que esses checks foram habilitados ou produzidos por outra ferramenta determinística habilitada.
  - IaC deve ser verificado por padrão com Checkov quando arquivos de IaC forem detectados deterministicamente.
  - Se nenhum arquivo de IaC for detectado, Checkov deve ficar como skipped e não deve reprovar.
  - IaC só deve ser desligado quando o usuário passar `--disable-iac` ou `--no-iac`.
  - Coverage NÃO é check fixo por padrão.
  - Coverage deve ser verificado somente quando o usuário passar `--enable-coverage`.
  - Quando coverage for habilitado, relatórios existentes devem ser lidos em formatos LCOV, Cobertura XML, JaCoCo XML, Clover XML ou Go coverprofile.
  - Se coverage for habilitado e nenhum relatório suportado for encontrado, o gate deve retornar evidência insuficiente.
  - O quality gate executável é determinístico; qualquer revisão semântica por agente/skill é complementar e fora do fluxo `quality-check .`.


Nova regra principal
--------------------

Substituir a ideia anterior de `quality-check .` como CLI ambíguo por esta regra:

  quality-check = wrapper local fino
  quality-sidecar = container Docker completo

Na prática:

  quality-check .

é equivalente a:

  docker run --rm \
    -v "<pasta-do-projeto>:/workspace" \
    -v "<pasta-do-projeto>/.quality/reports:/workspace/.quality/reports" \
    -w /workspace \
    code-approval-gates/quality-sidecar:latest \
    check /workspace

No PowerShell:

  docker run --rm `
    -v "${PWD}:/workspace" `
    -v "${PWD}/.quality/reports:/workspace/.quality/reports" `
    -w /workspace `
    code-approval-gates/quality-sidecar:latest `
    check /workspace


O que muda no plano anterior
----------------------------

Antes havia ambiguidade:

  - `quality-check .` poderia significar CLI local nativo.
  - `docker run ... quality-sidecar ...` poderia significar análise completa em container.

Agora não há ambiguidade:

  - `quality-check .` SEMPRE deve tentar rodar o container Docker.
  - a análise completa acontece dentro do container.
  - as ferramentas pesadas ficam dentro da imagem Docker.
  - a máquina local só precisa ter Docker disponível.


Definição final dos componentes
-------------------------------

1. quality-sidecar Docker image

Responsável por conter e executar:

  - MegaLinter
  - Semgrep
  - Checkov
  - Gitleaks
  - Trivy
  - OSV-Scanner
  - jscpd
  - ferramentas auxiliares
  - scripts de detecção de stack
  - detecção determinística de IaC
  - parser de relatórios de coverage quando habilitado
  - normalizador de relatórios
  - policy engine
  - cálculo de score
  - geração de quality-report.json
  - geração de quality-report.md

2. quality-check local wrapper

Responsável apenas por:

  - receber argumentos do usuário
  - descobrir a pasta alvo
  - criar a pasta `.quality/reports`
  - montar volumes no Docker
  - chamar `docker run`
  - repassar flags para o container
  - propagar exit code do container
  - imprimir erro amigável se Docker não estiver disponível

O wrapper local NÃO deve tentar reimplementar MegaLinter, Semgrep, Gitleaks, Trivy ou OSV nativamente.


Comportamento esperado
----------------------

Comando básico:

  quality-check .

Deve:

  1. validar se Docker está disponível;
  2. resolver o caminho absoluto da pasta atual;
  3. criar `.quality/reports` se não existir;
  4. executar o container `code-approval-gates/quality-sidecar:latest`;
  5. montar a pasta alvo em `/workspace`;
  6. rodar `check /workspace` dentro do container;
  7. gerar relatórios em `.quality/reports`;
  8. retornar o mesmo exit code do quality gate.


Exit codes finais
-----------------

O comando `quality-check .` deve retornar:

  0 = aprovado
  1 = reprovado pelo quality gate
  2 = precisa revisão humana / nenhuma análise suficiente / tool error relevante
  3 = erro operacional local, por exemplo Docker não instalado ou não iniciado


Flags preservadas
-----------------

O wrapper deve aceitar e repassar as flags para o container.

Exemplos:

  quality-check . --threshold 90
  quality-check . --profile strict
  quality-check . --allow-pii
  quality-check . --allow-secrets --allow-path "samples/**"
  quality-check . --enable-coverage --min-line-coverage 80
  quality-check . --disable-iac
  quality-check . --format=json,md
  quality-check . --output .quality/reports
  quality-check . --image code-approval-gates/quality-sidecar:dev


Flags específicas do wrapper
----------------------------

Estas flags pertencem ao wrapper local:

  --image <imagem>
      Define qual imagem Docker usar.
      Padrão: code-approval-gates/quality-sidecar:latest

  --pull
      Faz `docker pull` da imagem antes de executar.

  --no-pull
      Não tenta atualizar a imagem.

  --docker-arg <arg>
      Permite passar argumento adicional para `docker run`.

  --debug-docker
      Mostra o comando `docker run` completo antes de executar.


Flags repassadas ao container
-----------------------------

Estas flags são repassadas para o comando interno `check /workspace`:

  --threshold
  --profile
  --enable-pii
  --enable-secrets
  --disable-iac
  --no-iac
  --enable-coverage
  --coverage-report
  --min-line-coverage
  --min-branch-coverage
  --allow-pii
  --allow-secrets
  --allow-rule
  --allow-path
  --waiver
  --waiver-reason
  --waiver-expires
  --format
  --output
  --fail-on-tool-error
  --mode


Exemplo de implementação PowerShell
-----------------------------------

Arquivo: quality-check.ps1

  param(
    [Parameter(Position=0)]
    [string]$Target = ".",

    [string]$Image = "code-approval-gates/quality-sidecar:latest",
    [switch]$Pull,
    [switch]$NoPull,
    [switch]$DebugDocker,

    [Parameter(ValueFromRemainingArguments=$true)]
    [string[]]$RemainingArgs
  )

  $ErrorActionPreference = "Stop"

  function Fail($Message, $Code = 3) {
    Write-Error $Message
    exit $Code
  }

  $docker = Get-Command docker -ErrorAction SilentlyContinue
  if (-not $docker) {
    Fail "Docker não foi encontrado. Instale/inicie o Docker ou use uma alternativa compatível antes de rodar quality-check."
  }

  try {
    docker info *> $null
  } catch {
    Fail "Docker foi encontrado, mas não parece estar iniciado ou acessível. Abra/inicie o serviço Docker e tente novamente."
  }

  $TargetPath = Resolve-Path $Target
  $ReportsPath = Join-Path $TargetPath ".quality/reports"
  New-Item -ItemType Directory -Force -Path $ReportsPath | Out-Null

  if ($Pull -and -not $NoPull) {
    docker pull $Image
    if ($LASTEXITCODE -ne 0) { exit $LASTEXITCODE }
  }

  $dockerArgs = @(
    "run", "--rm",
    "-v", "${TargetPath}:/workspace",
    "-v", "${ReportsPath}:/workspace/.quality/reports",
    "-w", "/workspace",
    $Image,
    "check", "/workspace"
  )

  if ($RemainingArgs) {
    $dockerArgs += $RemainingArgs
  }

  if ($DebugDocker) {
    Write-Host "docker $($dockerArgs -join ' ')"
  }

  & docker @dockerArgs
  exit $LASTEXITCODE

Exemplo de wrapper Node.js opcional
-----------------------------------

Caso o projeto publique o CLI via npm, o binário `quality-check` pode ser um wrapper Node.js.

Arquivo: bin/quality-check.js

  #!/usr/bin/env node

  const { spawnSync } = require("child_process");
  const fs = require("fs");
  const path = require("path");

  const args = process.argv.slice(2);
  const target = args[0] && !args[0].startsWith("-") ? args.shift() : ".";

  const imageFlagIndex = args.indexOf("--image");
  let image = process.env.QUALITY_SIDECAR_IMAGE || "code-approval-gates/quality-sidecar:latest";

  if (imageFlagIndex >= 0) {
    image = args[imageFlagIndex + 1];
    args.splice(imageFlagIndex, 2);
  }

  const targetPath = path.resolve(target);
  const reportsPath = path.join(targetPath, ".quality", "reports");
  fs.mkdirSync(reportsPath, { recursive: true });

  const dockerInfo = spawnSync("docker", ["info"], { stdio: "ignore" });
  if (dockerInfo.status !== 0) {
    console.error("Docker não está instalado, iniciado ou acessível. O comando quality-check precisa do Docker para executar a análise completa.");
    process.exit(3);
  }

  const dockerArgs = [
    "run", "--rm",
    "-v", `${targetPath}:/workspace`,
    "-v", `${reportsPath}:/workspace/.quality/reports`,
    "-w", "/workspace",
    image,
    "check", "/workspace",
    ...args
  ];

  const result = spawnSync("docker", dockerArgs, { stdio: "inherit" });
  process.exit(result.status ?? 3);

Ajuste no Dockerfile do sidecar
-------------------------------

A imagem Docker deve continuar tendo o comando interno `quality-check` ou `check`.

Exemplo:

  ENTRYPOINT ["quality-check"]
  CMD ["check", "/workspace"]

Ou, se preferir separar:

  ENTRYPOINT ["/opt/quality-sidecar/entrypoint.sh"]
  CMD ["check", "/workspace"]

O entrypoint interno deve receber:

  check /workspace --threshold 90 --profile standard ...

E executar a análise completa dentro do container.


Novo fluxo final
----------------

O fluxo final passa a ser:

  usuário roda no PowerShell:

    quality-check .

  wrapper local executa:

    docker run --rm -v "projeto:/workspace" code-approval-gates/quality-sidecar:latest check /workspace

  container executa:

    MegaLinter
    Semgrep
    Checkov quando IaC for detectado
    Gitleaks
    Trivy
    OSV-Scanner
    jscpd
    build/test quando aplicável
    coverage quando explicitamente habilitado e relatório suportado existir
    normalização
    waivers
    score
    quality-report.json
    quality-report.md

  usuário recebe:

    APPROVED / NEEDS_CHANGES / REJECTED
    score
    lista de problemas
    relatórios em .quality/reports


Regras atualizadas para documentação
------------------------------------

Substituir qualquer trecho do plano anterior que diga ou sugira:

  "quality-check . pode rodar nativamente por padrão"

por:

  "quality-check . é um wrapper local container-first. Por padrão, ele chama Docker e executa a imagem quality-sidecar. A análise completa depende do Docker. Modos nativos podem existir no futuro, mas não são o comportamento padrão e não garantem paridade com o sidecar."

Substituir qualquer trecho que diga:

  "--engine auto tenta Docker, depois native"

por:

  "--engine docker é o padrão. Se Docker não estiver disponível, o comando falha com exit code 3. O modo native, se existir, deve ser uma opção explícita e documentada como parcial."


Política sobre modo nativo
--------------------------

O modo nativo pode existir no futuro, mas deve ser considerado parcial.

Se implementado, deve ser chamado explicitamente:

  quality-check . --engine native

E deve exibir aviso:

  "Modo native não garante paridade com o sidecar Docker. Algumas ferramentas podem ser puladas por ausência de instalação local. Para análise completa, use quality-check . com Docker disponível."

O comando básico:

  quality-check .

não deve cair silenciosamente para modo nativo, porque isso faria o usuário acreditar que recebeu a análise completa quando recebeu apenas uma análise parcial.


Conclusão da alteração
----------------------

A partir desta alteração:

  quality-check . = análise completa via Docker sidecar

Não é apenas uma CLI local.
Não é modo nativo por padrão.
Não é uma análise parcial.
É um alias/wrapper para rodar o container completo com todas as ferramentas empacotadas.
